title: "POST /message — JSON API v2 Send Message Endpoint"
sidebarTitle: "POST /message"
description: "POST /message sends an SMS, Viber, or WhatsApp message to up to 10 recipients. Accepts message body, flow, scheduling, and callback fields."
---

`POST /message` is the standard endpoint for sending a message to one or more recipients (up to 10). You can send a plain SMS by providing just a `body` and a `to` list, or define a multi-channel `flow` to attempt Viber or WhatsApp first and fall back to SMS automatically. The endpoint also supports scheduling, delivery-window restrictions, callbacks, and custom payload data that gets echoed back in webhook events.

## Endpoint

```text
POST https://api.smsmngr.com/v2/message
```

## Authentication

Pass your API key in the `x-api-key` request header:

```http
x-api-key: YOUR_API_KEY
```

## Request body parameters

<ParamField body="body" type="string" required>
  The message text, up to 1000 characters. This value is used when no channel-level `body` is defined inside `flow`. If every channel in your `flow` defines its own `body`, you may omit this field.
</ParamField>

<ParamField body="to" type="array" required>
  List of recipient objects. Must contain between 1 and 10 items. Each item must include a `phone_number`.

  <Expandable title="Recipient object fields">
    <ParamField body="phone_number" type="string" required>
      The recipient's phone number. International E.164 format without a leading `+` or `00` is recommended (e.g. `420777123456`). Other formats are accepted and automatically normalised.
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="flow" type="array">
  An ordered list of channel objects. SmsManager tries each channel in the order you specify — if the first channel is unavailable or the recipient cannot receive it, the next channel is attempted. Each item in the array must contain exactly one of: `sms`, `viber`, `whatsapp_text`, or `whatsapp_template`.
</ParamField>

<ParamField body="callback" type="string">
  A webhook URL that receives delivery status notifications for this message. If omitted, SmsManager uses your account's default callback URL (if configured). See the [Webhooks](/api-reference/json-v2/webhooks) reference for the event payload schema.
</ParamField>

<ParamField body="tag" type="string" default="promotional">
  A label used to group or categorise messages. Built-in values: `promotional` (default), `transactional`, `priority`. You can also pass any custom string to tag messages by campaign or use case.
</ParamField>

<ParamField body="datetime" type="string">
  Schedule the message for a specific future time. Must be an ISO 8601 datetime string in UTC, for example `2025-06-01T09:00:00Z`. Omit this field to send immediately.
</ParamField>

<ParamField body="delivery_time" type="object">
  Restrict delivery to a specific time window. Useful for ensuring messages are only delivered during business hours.

  <Expandable title="delivery_time fields">
    <ParamField body="days" type="array">
      Days of the week on which delivery is allowed. Accepted values: `monday`, `tuesday`, `wednesday`, `thursday`, `friday`, `saturday`, `sunday`.
    </ParamField>
    <ParamField body="start" type="string">
      Earliest allowed delivery time in `HH:MM` format, e.g. `08:00`.
    </ParamField>
    <ParamField body="end" type="string">
      Latest allowed delivery time in `HH:MM` format, e.g. `21:00`.
    </ParamField>
    <ParamField body="tz" type="string" default="UTC">
      IANA timezone name for `start` and `end`, e.g. `Europe/Prague`, `America/New_York`. Defaults to `UTC`.
    </ParamField>
  </Expandable>
</ParamField>

<ParamField body="payload" type="object">
  Arbitrary key-value data you want to associate with this message. SmsManager echoes this object back in every webhook event (delivery status, replies) so you can correlate notifications with your own records.

  ```json
  { "order_id": "ORD-1001", "user_id": "42" }
  ```
</ParamField>

<ParamField body="params" type="object">
  Special processing parameters, such as link shortening. Contact SmsManager support to learn how to use this field for your use case.
</ParamField>

## Flow channel fields

Each item in the `flow` array contains exactly one channel key. The fields available for each channel are described below.

### sms

<Expandable title="SMS channel fields">
  <ParamField body="body" type="string" required>
    The SMS message text (up to 1000 characters). Overrides the top-level `body` when present.
  </ParamField>
  <ParamField body="sender" type="string">
    Alphanumeric sender name (max 11 characters, pre-registration may be required) or a dedicated virtual number without a leading `+`, e.g. `420777123456`.
  </ParamField>
  <ParamField body="gateway" type="string" default="high">
    The SMS gateway to use. `high` is the standard shared gateway. Use `direct` if you are sending from a dedicated virtual number. Use `custom` if you use SIM hosting. Accepted values: `high`, `direct`, `custom`.
  </ParamField>
  <ParamField body="ttl" type="integer">
    Time to live in minutes. If the message cannot be delivered within this period, it is discarded.
  </ParamField>
  <ParamField body="type" type="string" default="utf">
    Character encoding mode. `utf` preserves Unicode characters (emoji, accented letters, non-Latin scripts). `sms` converts or removes Unicode to maximise the number of characters per SMS segment. Accepted values: `sms`, `utf`.
  </ParamField>
</Expandable>

### viber

<Expandable title="Viber channel fields">
  <ParamField body="body" type="string" required>
    The Viber message text (up to 1000 characters). Overrides the top-level `body` when present.
  </ParamField>
  <ParamField body="sender" type="string" required>
    The pre-registered Viber Business Message sender name.
  </ParamField>
  <ParamField body="buttons" type="array">
    Up to one call-to-action button displayed below the message. Each button object requires `title` (string) and `url` (URI string).
  </ParamField>
  <ParamField body="ttl" type="integer">
    Time to live in minutes. If the recipient cannot receive the Viber message within this period, the next channel in the flow is attempted (typically SMS fallback).
  </ParamField>
</Expandable>

### whatsapp_text

<Expandable title="WhatsApp text channel fields">
  <ParamField body="body" type="string" required>
    The WhatsApp message text (up to 1000 characters). You can only send a free-form body if the recipient has already replied to one of your template messages or messaged you first within the 24-hour service window.
  </ParamField>
  <ParamField body="sender" type="string" required>
    The Phone Number ID of your registered WhatsApp number. Find this in your [WhatsApp settings](https://app.smsmanager.com/whatsapp).
  </ParamField>
</Expandable>

### whatsapp_template

<Expandable title="WhatsApp template channel fields">
  <ParamField body="template_name" type="string" required>
    The name of a pre-approved WhatsApp message template (max 100 characters).
  </ParamField>
  <ParamField body="sender" type="string" required>
    The Phone Number ID of your registered WhatsApp number. Find this in your [WhatsApp settings](https://app.smsmanager.com/whatsapp).
  </ParamField>
  <ParamField body="language" type="string" required>
    The language code for the template, e.g. `en`, `cs`. You can have multiple language variants of the same template.
  </ParamField>
  <ParamField body="params" type="array">
    Array of string values for body placeholders in the template, in the order they appear.
  </ParamField>
  <ParamField body="params_header" type="array">
    Array of string values for header placeholders in the template.
  </ParamField>
  <ParamField body="params_buttons" type="array">
    Array of string values for button placeholders in the template.
  </ParamField>
  <ParamField body="ttl" type="integer">
    Time to live in minutes.
  </ParamField>
</Expandable>

## Response

A successful `200` response contains a `MessageResponse` object.

<ResponseField name="request_id" type="string">
  A unique identifier for this API request.
</ResponseField>

<ResponseField name="accepted" type="array">
  Recipients whose messages were accepted for delivery.

  <Expandable title="Accepted item fields">
    <ResponseField name="key" type="string">
      The zero-based index of the recipient in your `to` array (e.g. `"0"` for the first recipient).
    </ResponseField>
    <ResponseField name="message_id" type="string">
      Unique identifier for this recipient's message. Use this to correlate webhook delivery events.
    </ResponseField>
  </Expandable>
</ResponseField>

<ResponseField name="rejected" type="array">
  Recipients whose messages were rejected (e.g. invalid phone number, insufficient credit).

  <Expandable title="Rejected item fields">
    <ResponseField name="key" type="string">
      The zero-based index of the rejected recipient in your `to` array.
    </ResponseField>
  </Expandable>
</ResponseField>

## Examples

<CodeGroup>

```json Simple SMS
{
  "body": "Test message",
  "to": [
    { "phone_number": "420777123456" }
  ]
}
```

```bash Simple SMS — cURL
curl -X POST https://api.smsmngr.com/v2/message \
  -H "Content-Type: application/json" \
  -H "x-api-key: YOUR_API_KEY" \
  -d '{
    "body": "Test message",
    "to": [{ "phone_number": "420777123456" }]
  }'
```

</CodeGroup>

<CodeGroup>

```json Viber with SMS fallback
{
  "body": "Hello!",
  "to": [
    { "phone_number": "420777123456" }
  ],
  "flow": [
    {
      "viber": {
        "sender": "MySender",
        "body": "Hello!",
        "ttl": 60
      }
    },
    {
      "sms": {
        "sender": "MySender",
        "body": "Hello!"
      }
    }
  ]
}
```

```bash Viber with SMS fallback — cURL
curl -X POST https://api.smsmngr.com/v2/message \
  -H "Content-Type: application/json" \
  -H "x-api-key: YOUR_API_KEY" \
  -d '{
    "body": "Hello!",
    "to": [{ "phone_number": "420777123456" }],
    "flow": [
      { "viber": { "sender": "MySender", "body": "Hello!", "ttl": 60 } },
      { "sms": { "sender": "MySender", "body": "Hello!" } }
    ]
  }'
```

</CodeGroup>

## Error responses

| Status | Description |
|--------|-------------|
| `400 Bad Request` | The request body is invalid or missing required fields. The response body contains a `Message` string describing the problem, e.g. `"User is not authorized to access this resource with an explicit deny"`. Check that your `x-api-key` is correct and that all required fields are present. |
